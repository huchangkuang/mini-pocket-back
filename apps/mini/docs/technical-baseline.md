# 简易口袋技术基线文档（V1）

## 1. 技术栈与架构总览

- 框架：Taro 3.6 + React 18 + TypeScript。
- UI：Taro 原生组件 + `taro-ui`。
- 构建：Webpack5（Taro CLI）。
- 目录分层：
  - `src/pages`：页面与业务主逻辑。
  - `src/components`：复用展示组件。
  - `src/utils`：工具函数与平台 API 封装。
  - `src/hooks`：行为复用（节流/防抖）。
  - `config`：多端构建配置与环境差异。

## 2. 代码结构与模块关系

## 2.1 应用层

- `src/app.ts`：应用生命周期与热更新提示（`getUpdateManager`）。
- `src/app.config.ts`：页面注册、分包配置、窗口配置。

### 2.2 页面层（按能力域）

- 入口聚合：`pages/classify`。
- 互动能力：`handsBarrage`、`fingerUp`、`guessNumber`。
- 决策能力：`doDescription`。
- 工具能力：`qrcode`、`metronome`、`returnClock`、`lottery`、`beadArt`。

### 2.3 复用层

- 组件：`bomFixed`（底部固定容器）、`triangle`、`navBar`。
- Hooks：`useThrottle`、`useDebounce`。
- Store：`doDescription/store.ts`（本地常用决策项管理）。

### 2.4 工具层

- `wechatApi.ts`：支付、权限检查、Canvas 导出保存。
- `generateNum.ts`：随机数与号码生成。
- `errorToast.ts`：统一错误提示。
- `utils/code/*`：二维码/条码相关实现。

## 3. 平台 API 与第三方依赖边界

## 3.1 平台能力调用矩阵

| 能力       | 调用位置                       | 用途                 | 风险点                           |
| ---------- | ------------------------------ | -------------------- | -------------------------------- |
| 更新管理   | `src/app.ts`                   | 版本更新提示与重启   | 仅前台触发，更新失败缺少补偿提示 |
| 路由跳转   | 多页面                         | 页面流转与参数传递   | 参数过长/序列化失败              |
| 相册权限   | `utils/wechatApi.ts`、`qrcode` | 保存二维码图片       | 用户拒绝授权后体验中断           |
| Canvas     | `qrcode`、`beadArt`            | 图像/二维码绘制      | 大图与低端机性能问题             |
| 文件系统   | `beadArt`                      | Excel 文件写入与复制 | 存储空间与 IO 失败               |
| 音频上下文 | `metronome`                    | 节拍播放             | 中断恢复、机型兼容差异           |
| 分享       | 多页面                         | 页面分享卡片         | 分享路径需与页面注册一致         |

### 3.2 核心第三方依赖

- `exceljs`：拼豆能力导出 Excel 的核心依赖，关注大数据量内存占用。
- `dayjs`：时间处理（反方向时钟）。
- `classnames`：条件样式拼接。
- `taro-ui`：按钮、图标、滑块、弹窗等通用 UI。

## 4. 构建与发布基线

## 4.1 本地开发/构建命令

- 开发（微信）：`npm run dev:weapp`
- 打包（微信）：`npm run build:weapp`
- 其他端构建脚本已预置（h5/tt/qq/swan/alipay/jd/quickapp）

### 4.2 配置要点

- 别名：`@ -> src`（`config/index.js`）。
- 编译器：webpack5，预打包排除了 `taro-ui`。
- 样式：Sass + PostCSS（pxtransform、autoprefixer）。

### 4.3 发布前最小检查清单

1. 执行目标端构建命令并确认成功。
2. 核验分类页入口可达与核心功能可用（手持弹幕、做决定、二维码、节拍器、拼豆）。
3. 核验高风险能力（相册保存、Canvas 导出、音频播放）至少各 1 次。
4. 核验分享路径与页面注册一致。

## 5. 测试现状与质量约束

- 现状：自动化测试较少，当前可见单测主要在 `utils/generateNum`。
- 风险：页面逻辑以手工回归为主，跨页面回归成本高。
- 约束建议：
  - 新增核心能力至少提供 1 条可重复执行的冒烟测试说明。
  - 修改平台能力调用时必须附带失败场景验证记录。

## 6. 技术债与风险缓解

- 技术债 1：页面内业务逻辑偏重，状态与副作用耦合。
  - 缓解：逐步抽离可复用逻辑到 hooks 或 service 层。
- 技术债 2：部分页面存在隐式状态/候选模块（如隐藏入口、未注册页面）。
  - 缓解：明确保留与下线策略，并在配置层与文档层统一。
- 技术债 3：错误处理粒度较粗，失败重试与兜底策略不足。
  - 缓解：统一错误码/错误文案策略，补充关键流程兜底。

## 7. 文档更新规则

以下技术变更必须同步更新本技术文档：

1. 新增/删除页面、分包或入口映射。
2. 新增平台 API 调用或关键第三方依赖。
3. 构建配置、发布流程、质量门槛发生变化。

建议在每次需求迭代中，由开发负责人在合并前完成文档差异核对。
